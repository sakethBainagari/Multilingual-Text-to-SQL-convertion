# ⚙️ Backend Documentation - text2sqlAgent

## Overview

The backend is a Python-based server that handles natural language processing, AI-powered SQL generation, database operations, and serves the web interface. It's the brain of the application.

---

## Tech Stack

### 1. Python

**What it does in this project:**
- Serves as the core programming language for all backend logic
- Provides native SQLite support for database operations
- Handles string processing and regex for query parsing
- Manages file I/O for database uploads and FAISS persistence
- Orchestrates communication between all components (AI models, database, cache)

**Why Python was chosen:**
- Rich ecosystem for AI/ML libraries (Sentence Transformers, FAISS)
- Excellent support for web development (Flask)
- Easy integration with Google's Gemini API
- Native JSON handling for API responses
- Strong community support for NLP tasks

---

### 2. Flask

**What it does in this project:**

#### Web Server
- Runs the HTTP server on port 5000
- Serves the HTML frontend from the `templates/` folder
- Handles incoming HTTP requests (GET, POST)
- Returns JSON responses for API endpoints

#### REST API Framework
- Defines all API endpoints (routes) for the application
- Parses incoming JSON request bodies automatically
- Handles file uploads for database files
- Manages HTTP response codes and error handling

#### Template Rendering
- Serves `index.html` as the main web interface
- Flask's template engine loads HTML from `templates/` folder

#### CORS (Cross-Origin Resource Sharing)
- Flask-CORS extension enables the frontend to make API calls
- Required for browser security when frontend and backend communicate

**Endpoints Created:**
- `/` - Serve web interface
- `/api/query` - Main query processing
- `/api/similarity-check` - Cache lookup
- `/api/generate-sql` - LLM SQL generation
- `/api/execute-sql` - Direct SQL execution
- `/api/db/*` - Database management
- `/api/export` - Data export

---

### 3. SQLite

**What it does in this project:**

#### Primary Database Engine
- Stores all application data (employees, departments, projects tables)
- Executes SQL queries generated by AI models
- Returns query results as rows and columns

#### Schema Discovery
- Provides table structure information via `PRAGMA table_info()`
- Lists all tables via `sqlite_master` system table
- Used to build context for AI prompts

#### Multi-Database Support
- Default database: `data/advanced_nlsql.db`
- Uploaded databases stored in: `data/uploads/`
- Supports switching between databases at runtime

#### Thread Safety
- Uses `check_same_thread=False` for Flask's multi-threaded environment
- Opens new connections per request to avoid conflicts

**Database Operations:**
- SELECT queries return Pandas DataFrames
- INSERT/UPDATE/DELETE return affected row counts
- CREATE TABLE for dynamic schema creation
- Transaction management with commit/rollback

### 4. Google Gemini 2.0 Flash (Online AI)

**What it does in this project:**

#### SQL Query Generation
- Receives natural language query + database schema
- Understands user intent (show, filter, aggregate, etc.)
- Generates syntactically correct SQLite queries
- Handles complex queries (JOINs, GROUP BY, subqueries)

#### Multilingual Understanding
- Processes queries in English, Telugu, and Hindi
- Maps non-English terms to SQL keywords
- Understands numeric values in any language

#### Context-Aware Generation
- Receives full database schema in prompt
- Uses exact table and column names
- Applies appropriate data types in conditions

**How it works:**
1. Backend formats schema into readable text
2. Creates detailed prompt with instructions and examples
3. Sends prompt to Gemini API
4. Receives SQL query as response
5. Cleans response (removes markdown, extra text)
6. Validates SQL structure before execution

**Configuration:**
- Low temperature (0.1) for deterministic output
- Max 512 output tokens
- Retry logic with exponential backoff for rate limits

---

### 5. Ollama + TinyLlama 1.1B (Offline AI)

**What it does in this project:**

#### Offline SQL Generation
- Works without internet connection
- Runs locally on user's machine
- Provides same functionality as Gemini but offline

#### Local LLM Server
- Ollama runs as a background service on localhost:11434
- Hosts the TinyLlama 1.1B model (small but capable)
- Responds to HTTP API requests

#### Fallback Mechanism
- If Gemini quota exhausted, switches to Ollama
- If Ollama unavailable, uses rule-based fallback
- Ensures system always provides a response

**How it works:**
1. Backend checks if Ollama is running (health check)
2. Sends prompt to Ollama's `/api/generate` endpoint
3. Ollama runs inference on TinyLlama model
4. Returns generated SQL query
5. Same cleaning and validation as Gemini

**Trade-offs:**
- Slower than Gemini (runs on CPU)
- Less accurate for complex queries
- No API costs or rate limits
- Works in air-gapped environments

### 6. Sentence Transformers

**What it does in this project:**

#### Text Embedding Generation
- Converts natural language queries into numerical vectors
- Uses `paraphrase-multilingual-MiniLM-L12-v2` model
- Produces 384-dimensional embeddings

#### Multilingual Support
- Same model handles English, Telugu, Hindi queries
- Embeds semantically similar queries close together
- Cross-lingual similarity (English query matches Telugu equivalent)

#### Query Normalization
- Replaces numbers with `<NUM>` placeholder before embedding
- Allows matching "salary > 50000" with "salary > 70000"
- Focuses on query structure, not specific values

**How it works:**
1. User query comes in: "Show employees with salary > 50000"
2. Normalized: "Show employees with salary > <NUM>"
3. Encoded to 384-dimensional vector
4. Vector used for FAISS similarity search

**Why this model:**
- Supports 50+ languages
- Compact size (fast inference)
- Good balance of accuracy and speed
- Open source and free

---

### 7. FAISS (Facebook AI Similarity Search)

**What it does in this project:**

#### Vector Database
- Stores embeddings of all previously successful queries
- Enables fast similarity search across thousands of queries
- Uses L2 (Euclidean) distance for similarity measurement

#### Query Caching
- When new query comes in, searches for similar cached queries
- If similarity > 70%, returns cached SQL (no LLM call needed)
- Reduces response time from seconds to milliseconds

#### Persistence
- Saves index to `data/query_cache.faiss` binary file
- Saves metadata to `data/query_metadata.json`
- Loads on startup, saves after each new query

**How similarity search works:**
1. New query embedded to 384-dim vector
2. FAISS searches k-nearest neighbors (k=5)
3. Calculates similarity score: `1 / (1 + distance)`
4. Returns matches above 70% threshold
5. Best match's SQL used with entity swapping

#### Auto-Learning
- Every successful LLM-generated query gets added to FAISS
- System becomes smarter over time
- Common queries eventually all served from cache

---

### 8. Entity Swapping System

**What it does in this project:**

#### Adapting Cached SQL
- When similar query found, numbers/strings might differ
- Extracts entities from original and new query
- Replaces values in cached SQL with new values

**Example:**
```
Cached: "Show employees with salary > 50000"
    SQL: SELECT * FROM employees WHERE salary > 50000;

New: "Show employees with salary > 70000"
    Adapted SQL: SELECT * FROM employees WHERE salary > 70000;
```

#### What gets swapped:
- Numeric values (50000 → 70000)
- Quoted strings ('Engineering' → 'Marketing')
- Date values ('2023' → '2024')

#### Structural Change Detection
- Detects when query structure differs too much
- If different columns, aggregations, or operations requested
- Falls back to LLM generation instead of swapping

## Frontend-Backend Connection

### How They Communicate

#### Request Flow
1. **User Action**: Types query and clicks submit
2. **JavaScript**: Captures form data, creates JSON object
3. **Fetch API**: Sends HTTP POST to Flask endpoint
4. **Flask**: Receives request, parses JSON body
5. **Processing**: Similarity check → LLM → SQL execution
6. **Response**: Flask returns JSON with results
7. **JavaScript**: Parses JSON, updates DOM
8. **User Sees**: Results table and visualization

### Data Format

**Request (Frontend → Backend):**
```json
{
  "query": "Show all employees in Engineering",
  "visualize": true,
  "chart_type": "bar"
}
```

**Response (Backend → Frontend):**
```json
{
  "success": true,
  "sql_query": "SELECT * FROM employees WHERE department = 'Engineering';",
  "data": [{"name": "John", "department": "Engineering", "salary": 75000}],
  "row_count": 3,
  "execution_time": 0.045,
  "visualization_data": {...},
  "cache_hit": false
}
```

### File Uploads

**Database Upload Flow:**
1. User selects .db file in modal
2. JavaScript creates FormData with file
3. POST to `/api/db/upload` with multipart encoding
4. Flask saves file to `data/uploads/`
5. Validates SQLite structure
6. Switches active database
7. Returns success with table list

## Backend Processing Pipeline

### Step-by-Step Query Processing

```
1. RECEIVE QUERY
   ↓
2. SIMILARITY CHECK (FAISS)
   ├── Cache Hit (>70% similar) → Entity Swap → Go to Step 4
   └── Cache Miss → Continue to Step 3
   ↓
3. LLM SQL GENERATION
   ├── Try Gemini (online) → Got SQL → Continue
   ├── Try Ollama (offline) → Got SQL → Continue
   └── Rule-based fallback → Basic SQL → Continue
   ↓
4. SQL EXECUTION (SQLite)
   ├── SELECT → Return DataFrame
   └── INSERT/UPDATE/DELETE → Return affected rows
   ↓
5. VISUALIZATION (if requested)
   ├── Auto-detect chart type
   └── Generate Plotly JSON
   ↓
6. AUTO-LEARN (save to FAISS)
   ↓
7. RETURN JSON RESPONSE
```

## Error Handling

| Error Type | Handling |
|------------|----------|
| Empty query | Return error message immediately |
| LLM timeout | Retry with backoff, then fallback |
| SQL syntax error | Return error with details |
| Database not found | Switch to default database |
| FAISS not loaded | Skip similarity, use LLM only |
| File upload invalid | Validate and return specific error |
